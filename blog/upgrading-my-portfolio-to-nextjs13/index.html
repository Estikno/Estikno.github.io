Upgrading my Portfolio to Nextjs13

Authors
  • Name
    David
    Twitter
    @Estikno

Introduction

Hey there, fellow developers and visitors! I’m excited to share my recent adventure of upgrading my portfolio from Next.js 12 to the shiny new Next.js 13. This journey wasn’t just about adopting the latest technology but also about reimagining my portfolio’s structure and adding a brand-new blog system using MDX. Join me as I walk you through the process, the challenges I faced, and the triumphs that emerged.

The Motivation Behind the Upgrade

A few months back, I stumbled upon the promising features of Next.js 13, from improved performance to snappy developer experience.

Also one of the main factor of waiting to make the transition was to await the stable release of the /app directory. As it seemed to be an excellent choice for the project and also a good oppornity to learn new things.

I knew it was time to embrace this upgrade and take advantage of the goodies it had to offer. Not to mention that the sooner I made the change to the newer version, the easier it would be to mantain the project in the latest version of Next.js.

Considering Alternatives: Astro and Sveltekit:

During my upgrade deliberations, I considered alternatives like Astro and Sveltekit. Both offered intriguing possibilities, but my deep-rooted fondness for React and my solid grasp of it made me lean towards sticking with it. Moreover, as I delved into these alternatives, the potential hurdles of using Astro with dynamic routes gave me pause and reaffirmed my decision to proceed with Next.js, even thought transitioning to the newer version would not be easy, specially with the new /app directory.

However, with Astro’s reported compatibility with React as an alternative template engine made me realize that if I were to recreate my portfolio today, I would probably use Astro. Not to mention its excellent performance. So do not be surprised if I make a post talking about it and trying it.

Restructuring the Project

First order of business? Rethinking the project structure. I decided to bid farewell to the traditional /pages directory and welcome the /app structure with open arms. This new approach brought clarity to my codebase, making it easier to manage different sections of my portfolio without the clutter.

However, the transition wasn’t without its share of challenges. Lot of code had to be changed as the new directory required a new structure of code, as well as implementing new things and removing others. So a lot had to be changed.

Embracing MDX for Blogging

Ah, the thrill of adding a blog section! I chose MDX for its magical combination of Markdown’s simplicity and React’s power, althought plain MD also works if it is needed. Setting it up was a journey of its own, from integrating MDX into my project to crafting my first blog post with rich interactive elements.

And here’s where Contentlayer entered the scene, making the integration of MDX even more seamless. This nifty tool abstracted away much of the complex setup, allowing me to focus on creating engaging content without battling the intricacies of MDX integration, such as layouts, file structures and more. The blog instantly felt like an integral part of my portfolio, adding depth and context to my work.

Transitioning to Next.js 13: New Syntax Challenges

The upgrade to Next.js 13 wasn’t a walk in the park, mainly due to the syntax changes that had me scratching my head a few times. API routes, file extensions, server and client objects, and route configurations demanded a closer look and a fair share of trial and error.

The Next.js documentation became my reliable companion during this phase, along with numerous Google searches and assistance from Stack Overflow. Remember, challenges are just opportunities to learn, and learn I did!

Take as an example this old Nextjs 12 code of the blogs page:

blog.ts
export async function getStaticProps() {
    const posts = await getAllFilesFrontMatter('blog')
    const initialDisplayPosts = posts.slice(0, POSTS_PER_PAGE)
    const pagination = {
        currentPage: 1,
        totalPages: Math.ceil(posts.length / POSTS_PER_PAGE),
    }

    return { props: { initialDisplayPosts, posts, pagination } }
}

export default function Blog({ posts, initialDisplayPosts, pagination }) {
    return (
        <>
        <PageSEO title={`Blog - ${siteMetadata.author}`} description={siteMetadata.description} />
        <ListLayout
            posts={posts}
            initialDisplayPosts={initialDisplayPosts}
            pagination={pagination}
            title="All Posts"
        />
        </>
    )
}

An this is how it looks right now with Nextjs 13:

page.tsx
export const metadata = PageSEO({
    title: `Blog - ${siteMetadata.author}`,
    description: siteMetadata.description,
    pathname: "/blog"

const getStaticProps = async () => {
    const posts = sortedBlogPost(allBlogs);
    const initialDisplayPosts = posts.slice(0, POSTS_PER_PAGE);
    const pagination = {
        currentPage: 1,
        totalPages: Math.ceil(posts.length / POSTS_PER_PAGE),
    };

    return {

            pagination,
        },
    };
};

export default async function Blog() {
    const {
        props,
    }: { props: { initialDisplayPosts: any; posts: any; pagination: any } } =
        await getStaticProps();
    return (
        <ListLayout
            posts={props.posts}
            initialDisplayPosts={props.initialDisplayPosts}
            pagination={props.pagination}
            title="All Posts"
        />
    );
}

Now, this might not seem like a huge change, and on its own, it is not. However, consider having to change dozens of little scripts like this one and make sure that everything works fine, then it is more complicated.

Lessons Learned and Takeaways

Looking back, this upgrade taught me a lot. It’s not just about the technology but also about resilience and curiosity. Adopting new paradigms takes time, patience, and a willingness to dive deep. The revamped structure brought a sense of order to my portfolio, and the blog infused it with personality. It was also my first time implementing a blog, so it was very amazing seeing in the end that it worked amazingly well.

Conclusion

And there you have it—my journey from Next.js 12 to 13, from /pages to /app, and from a simple portfolio to a dynamic one with a blog. It wasn’t just about lines of code; it was about evolution, growth, and embracing the unknown. I invite you to explore my upgraded portfolio, delve into the blog posts, and join me in celebrating the ever-evolving world of web development.

If you’re currently on Next.js 12, I highly recommend considering the upgrade to version 13 to take advantage of its enhanced features and improvements, and do not forget to transition to the new /app directory, despite the many changes you will need to make it is worth it in the end.
